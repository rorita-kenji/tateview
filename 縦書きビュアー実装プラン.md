# 1. 目的とゴール

v5仕様を満たす縦書き小説ビューアを、単一HTMLで動作する形で実装する。最終ゴールは次の3点を同時に成立させることにある。

- 文字数ベースの決定的なページ分割（ウィンドウサイズに依存しない）
- 20万文字で快適、100万文字でベストエフォートの性能
- 公募原稿の校正に使えるワーニング検出と位置復帰の精度

---

# 2. 技術スタック方針

素のHTML/CSS/JavaScript（ES Modules）で構成し、外部ライブラリは使わない（依存ゼロ）。縦書きはCSSの `writing-mode: vertical-rl` を基盤とする。重い処理（文字コード判定・正規化・トークナイズ・ページ分割・ワーニング検出・検索）はすべて Web Worker 側に隔離し、メインスレッドは描画とUI操作に専念する。

開発は複数モジュールで行い、最終的に `node build.js` が **main用とworker用の2バンドル**を明示的に結合して単一HTML化する。Worker は結合後の worker バンドルを Blob URL でインライン生成する。文字コード判定は自前の簡易判定（BOM検出＋UTF-8妥当性検査＋Shift_JIS推定）で行い、デコードは `TextDecoder` を用いる。

---

# 3. アーキテクチャ

処理は「メインスレッド」と「Worker」の二層に分ける。

メインスレッドは、ファイル読み込みトリガー、UI状態管理、縦書きDOMのレンダリング（表示ページのみ）、ナビゲーション、選択・コピー、設定の保存/復元を担う。

Worker は、生バイト列の文字コード判定とデコード、改行正規化、トークナイズ（ストリーミング）、ページ分割、ワーニング検出、全文検索を担う。分割結果は「ページ境界（各ページ・各列の半開区間 `[start,end)` UTF-16）」という軽量な配列としてメインスレッドへ返し、本文文字列そのものはメイン側でも保持する（描画のため）。全Token配列は保持しない。

Worker通信は `documentId`（原稿を開くたびに採番）と `requestId`（リクエストごとに単調増加）を付け、main側は最新ID以外の結果を破棄する。長いループ中は数千文字ごとにイベントループへ制御を返し、古いジョブを途中キャンセルできるようにする。

```
[File] → main → Worker(判定/正規化/トークナイズ(gen)/分割/警告/検索)
                   │  ページ境界(半開区間)・警告・検索結果（documentId/requestId付き）
                   ▼
        main（表示ページのみDOM生成 / 縦書き描画 / ナビ / 選択 / 古い結果を破棄）
```

# 4. データモデル

中核は「正規化済み本文文字列」と「ページ境界（半開区間の配列）」である。

正規化済み本文は、改行をLFへ揃えた単一文字列とする。**オフセットは正規化済み文字列上の UTF-16 コードユニット位置**で一貫して扱い、現在位置保持・検索ヒット・ワーニング位置のすべてがこの基準。範囲は半開区間 `[start, end)` に統一し、`text.slice(start, end)` で範囲が復元できることを不変条件とする。文字の走査は `for...of`（コードポイント単位）で行い、カーソルは `char.length`（BMP=1・サロゲートペア=2）だけ進める。

トークンは、全角文字・全角スペース（各1マス）、半角トークン（連続する半角列を1トークン、`ceil(半角文字数 × 0.5)` マス）、**ルビ（複合トークン `{type:'ruby', baseText, rubyText, sourceStart, sourceEnd, mass}`、分割不能、マスにカウントしない）**、改行（強制改列、`sourceEnd`まで直前列に含める）に分類する。トークンは generator で逐次生成し、配列化しない。

ページ境界は各ページについて `{ index, range:{start,end}, columns: {start,end}[] }` を持つ配列とする。ページ→本文範囲、本文オフセット→ページの相互変換（二分探索）が可能になり、仮想描画と位置復帰が成立する。

# 5. モジュール構成

| モジュール | 責務 | 配置 |
|-----------|------|------|
| encoding | BOM/UTF-8/Shift_JIS 判定・デコード | Worker |
| normalize | 改行正規化（CRLF/CR→LF） | Worker |
| tokenizer | 全角/半角/ルビ(複合)/改行のトークン化（generator） | Worker |
| paginator | 文字数＋禁則＋ぶら下げによるページ境界算出（半開区間） | Worker |
| warnings | 22項目のワーニング検出（severity・ラン集約。全角英数・絵文字の info 含む） | Worker |
| search | 全文・見出し検索 | Worker |
| renderer | 縦書きDOM生成（表示ページのみ）・コピー正規化 | main |
| navigator | キー/ホイール/スワイプ/ジャンプ/サムネイル | main |
| position | 文字オフセット＋周辺文字列（before/after両一致）による復帰 | main |
| settings | LocalStorage保存・復元（位置キーはファイル名基準） | main |
| filewatch | File System Access API 更新確認・手動更新 | main |
| ui | テーマ・フォント・パネル・警告一覧UI | main |

`modules/` は副作用のない純粋関数として書き、Node からも import してテストできるようにする。

# 6. フェーズ別工程

監査提案の順序を採用する。各フェーズは単体で動作確認できる状態を成果物とし、該当テストを通してから次へ進む。

## フェーズ0：土台

単一HTMLの雛形、Worker通信の疎通（documentId/requestId付き）、ファイル選択→Workerへバイト列受け渡しまでを通す。受け入れ基準は、ファイルを開くとWorkerで文字数がカウントでき、古い結果が破棄されること。

## フェーズ1：入力パイプライン

文字コード自動判定（BOM/UTF-8/Shift_JIS）、判定失敗時の手動選択UI、改行正規化を実装する。受け入れ基準は、UTF-8/Shift_JISの両サンプルを文字化けなく読み込め、判定失敗時に選択で復旧できること。

## フェーズ2：縦書き表示スパイク（実機確認先行）

20〜30文字だけの最小縦書き表示を作り、半角英数・半角カナ・ルビ・複数列をまたぐ選択・コピーを Chrome/Edge/Firefox/Safari で実機確認する。`text-orientation: sideways` の効き、コピー結果が親文字のみになること（copyイベント正規化）を確認する。ここでブラウザ差の地雷を早期に踏んでおく。

## フェーズ3：ページ分割エンジン（最重要）

ストリーミング tokenizer と paginator を実装する。まず禁則OFF（機械分割）で文字数ベースの決定的分割を成立させ、次に禁則ON（最後の有効境界探索方式）、ぶら下げ、ルビの非カウントを順に載せる。半角トークンは `ceil(n×0.5)` マス、分割不能トークン（半角ラン・ルビ親文字）が列容量を超える場合はオーバーフロー列として単独確定（空列を無限生成しない）。改行は直前列に含める。受け入れ基準は、同一原稿・同一設定で分割結果が常に一致し、半開区間の不変条件（隙間/重複なし・最終end===length）を満たすこと。

## フェーズ4：本UI・レンダリングと仮想描画

`writing-mode: vertical-rl` で表示ページ（＋前後プリフェッチ）のみDOM生成する。半角の横倒し（sideways）、ルビ表示、見出しの色・太字（組版は本文と同一）、選択・コピー正規化を実装する。ウィンドウリサイズ時はページ内容・境界を変えずフォントサイズ・行間・セルサイズのみ調整する。受け入れ基準は、リサイズでページ境界が動かず、コピー結果にルビが混入しないこと。

## フェーズ5：ナビゲーションと位置保持

左右キー/スペース/ホイール/スワイプ、ページジャンプ、サムネイルドラッグとミニプレビューを実装する。現在位置は文字オフセット＋周辺文字列（before/after両一致スコアリング）で保持し、位置キーはファイル名基準にする。受け入れ基準は、文字数が変わる編集後も位置情報を再利用して近傍へ復帰できること。

## フェーズ6：組版オプションと設定保存

禁則・ぶら下げ・半角色表示のON/OFF切替、禁則文字一覧の編集、テーマ・フォント種別・サイズ変更、LocalStorageへの保存/復元を実装する。**ルビ表示ON/OFFはページ境界を変えず再描画＋ruby-off警告の再検出のみ**行う（再ページ分割しない）。受け入れ基準は、設定変更でページ境界が仕様どおりに扱われ、再起動で全設定が復元されること。

## フェーズ7：ワーニング

15項目の検出をWorkerで実装し、severity・ラン集約・見出しプレフィックス除外・上限＋総件数表示を用意する。一覧表示・該当箇所ジャンプ・強調表示・前後移動、サムネイル/スクロールバーへのマーカーも行う。受け入れ基準は、既知の誤り原稿で各項目が検出され、`# 見出し`が不要な半角警告を出さないこと。
（v1.0 時点の確定は **22** 項目。全角英字／全角数字／絵文字は info。校正パネル上で全角英字・全角数字の検出をON/OFF可。詳細は `docs/校正ワーニング一覧.md`。）

## フェーズ8：検索

全文検索（件数・前後移動・強調）と見出し検索を実装する。大容量でも操作性を保つため非同期実行＋大量ヒット時の上限/総件数表示とする。受け入れ基準は、20万文字で検索が体感的に待たされないこと。

## フェーズ9：ファイル更新確認・ビルド・仕上げ

File System Access API 対応環境での更新確認と再読み込み（現在位置保持）、非対応・`file://` 向け手動更新ボタン（再選択）、初回分割の進捗表示、`build.js`（2バンドル）による単一HTML化、対応ブラウザ横断確認を行う。**AIチェックのUI・APIキー入力欄は今回作らない**（スコープ外）。

# 7. 技術リスクと対策

最大のリスクはページ分割の決定性である。DOM計測に依存すると環境差でページがずれるため、分割は文字数モデル（半角0.5マス・切り上げ）だけで完結させ、DOMは表示専用と割り切る。範囲は半開区間で管理し不変条件をテストで守る。

次に大容量時のメモリと性能。100万文字を1文字1オブジェクトにすると原文よりはるかに大きなメモリを消費するため、tokenizer は generator でストリーミングし、paginator はページ境界だけを保持する（全Token配列を持たない）。設定変更時はWorker内で再走査する。チャンク処理（5000文字目安）と進捗通知、途中キャンセルでUIを固めない。

縦書き＋ルビ＋半角横倒しでのテキスト選択・コピーはブラウザ差が出やすい。フェーズ2で先に実機確認する。半角は `text-orientation: sideways` を明示し、コピーは `copy` イベントで親文字のみに正規化する。

Worker結果の追い越しリスクは documentId/requestId の世代管理で防ぐ（古い結果は破棄）。

File System Access API は Safari/Firefox 非対応のため、対応環境のみ更新確認、非対応・`file://` は手動ボタンへフォールバックする。「自動リロード」は実機挙動差が大きいため控えめに扱う。

# 8. テスト方針

ページ分割は最重要のため、代表原稿（プリセット各種・禁則境界・半角混在・長いルビ・連続約物）で期待ページ境界を固定した回帰テストに加え、ランダム入力による不変条件（プロパティ）テストを組む。tokenizer/paginator は generator/純粋関数として Worker 外でも単体テストする（配列化しないことも確認）。

加えて最低限、次のケースを追加する。

- `A😀B` の `sourceStart/sourceEnd` が UTF-16 位置になる／全オフセットで `text.slice(start,end)` が対応範囲を返す
- ルビ親文字の途中に列境界が入らない／ルビON/OFFで `Page[]` が完全一致
- 列幅より長い半角ランで空列が生成されない／列幅より長いルビ親文字で処理が停止しない
- 先頭改行・連続改行・末尾改行・空ファイル（ページ数と表示）
- 行頭/行末禁則文字の連続／ぶら下げ直後の改ページ
- 古い documentId/requestId の結果が画面を更新しない
- 文字数が変わる編集後も位置情報を再利用できる
- ビルド後HTMLに import文・外部script・外部stylesheet が残らない／Blob Worker が実際に起動する
- ルビ付き範囲のコピー結果にルビ文字列が含まれない

ワーニングは各項目「検出されるべき例／されてはいけない例」を対で検証する。性能は20万・100万文字（日本語のみ／ASCIIのみ）で初回分割時間・ページ送り・検索・大量警告候補を計測する。最後に対応ブラウザ（Chrome/Edge/Firefox/Safari、iPhone/Android）で横断確認する。
